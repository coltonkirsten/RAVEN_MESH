# RAVEN_MESH Operational Playbook

**Audience:** operator running RAVEN_MESH in production.
**Scope:** prescriptive runbooks for the six most common operational situations.
**Layer convention:** every step is tagged `[PROTOCOL]` (Core, supervisor, manifest, audit log, admin API) or `[OPINIONATED]` (specific nodes, dashboard, run scripts). Treat the protocol layer as the moat — it must stay generic. Opinionated steps may be replaced wholesale if the deployment swaps node implementations.

This playbook assumes:
- Core is launched with `--supervisor` enabled (commit `71f70e8`). The supervisor admin endpoints (`/v0/admin/spawn|stop|restart|reconcile|processes`) require this. Without `--supervisor`, those endpoints return `409 supervisor_disabled` and `scripts/run_mesh.sh` owns process lifecycle instead.
- `ADMIN_TOKEN` is set in the environment of the Core process. All admin endpoints require `X-Admin-Token: $ADMIN_TOKEN` (or `?admin_token=…`). Do not run production without overriding `admin-dev-token`.
- Every shell example uses `MESH_HOST=127.0.0.1 MESH_PORT=8000` as defaults; substitute your own.

## 1. Cold start procedure

A cold start brings the entire mesh up from a stopped state. Two paths exist; pick one and stick with it for a given deployment. Mixing them creates duplicate processes.

### Path A (recommended): Core-supervised cold start `[PROTOCOL]`

This is the path commit `71f70e8` was built for. Core owns every node process.

1. **Verify host preconditions.**
   - `python3 --version` ≥ 3.11.
   - The repo working tree is clean and on the deploy ref. `git rev-parse HEAD` should match the build manifest.
   - `.logs/` and `.pids/` directories are writable by the deploy user.
   - Disk has at least 2× the size of `audit.log` free, plus headroom for per-node logs.

2. **Inspect the manifest.** `[PROTOCOL]`
   - `python3 -c 'import yaml,sys; yaml.safe_load(open(sys.argv[1]))' manifests/<deploy>.yaml`
   - Every `node.id` referenced from `relationships` must exist as a node.
   - Every `surfaces[].schema` path must resolve relative to the manifest file.
   - Every `identity_secret: env:NAME` requires `NAME` to be present in the environment of the Core process — do **not** rely on Core's autogen fallback in production; autogenerated secrets are not persisted across restarts and will silently break HMAC signing for any node that already has a different secret cached.

3. **Export required environment.**
   ```sh
   export ADMIN_TOKEN="$(pass raven-mesh/admin-token)"
   export MESH_MANIFEST=manifests/<deploy>.yaml
   export MESH_HOST=127.0.0.1
   export MESH_PORT=8000
   export AUDIT_LOG=/var/lib/raven-mesh/audit.log
   # plus every per-node identity_secret env var named in the manifest
   ```

4. **Launch Core with the supervisor and auto-reconcile.** `[PROTOCOL]`
   ```sh
   python3 -m core.core \
     --manifest "$MESH_MANIFEST" \
     --host "$MESH_HOST" --port "$MESH_PORT" \
     --audit-log "$AUDIT_LOG" \
     --supervisor \
     --supervisor-log-dir /var/log/raven-mesh \
     --auto-reconcile
   ```
   `--auto-reconcile` makes Core call `supervisor.reconcile(manifest_nodes)` 200 ms after the listener comes up, spawning every manifest node whose `runner` is resolvable.

5. **Verify the listener.** `[PROTOCOL]`
   ```sh
   curl -fsS http://$MESH_HOST:$MESH_PORT/v0/healthz
   # expect: {"ok":true,"nodes_declared":N,"nodes_connected":≤N,"edges":E,"pending":0}
   ```

6. **Verify supervisor children.** `[PROTOCOL]`
   ```sh
   curl -fsS -H "X-Admin-Token: $ADMIN_TOKEN" \
     http://$MESH_HOST:$MESH_PORT/v0/admin/processes
   ```
   Every child should be `status: running` with non-zero `pid`. Anything in `failed` exhausted its restart budget — read its log under `--supervisor-log-dir/<node_id>.log` before retrying.

7. **Verify connectivity, not just liveness.** `[PROTOCOL]` + `[OPINIONATED]`
   `nodes_connected` (from `/v0/healthz`) should equal the count of long-lived nodes in the manifest. A running pid that has not registered is a configuration mismatch — typically a stale `identity_secret` env var, or the node's runner script not pointing Core at `$MESH_HOST:$MESH_PORT`.

8. **Smoke a known-safe invocation.** `[OPINIONATED]`
   Use whichever no-op invocation the deployment has. For demo-style deployments, `tasks.list` returning `[]` works:
   ```sh
   curl -fsS -H "X-Admin-Token: $ADMIN_TOKEN" \
     -H "Content-Type: application/json" \
     -d '{"from_node":"<some_node>","target":"tasks.list","payload":{}}' \
     http://$MESH_HOST:$MESH_PORT/v0/admin/invoke
   ```
   This synthesizes a Core-signed envelope, exercises edge ACLs, schema validation, and SSE delivery in one round trip.

### Path B (legacy): script-driven cold start `[OPINIONATED]`

Use this only when `--supervisor` is intentionally disabled.

1. `scripts/run_mesh.sh manifests/<deploy>.yaml`
2. Core admin process endpoints will return `409 supervisor_disabled`. Ownership of process lifecycle lives in the shell script and `.pids/`.
3. To stop: `scripts/run_mesh.sh stop`.

Do not use both paths in the same deployment. The script writes pidfiles the supervisor doesn't know about, and the supervisor spawns subprocesses the script can't see.

## 2. Adding a new node

There are two surfaces you can add a node from. The dashboard surface is **opinionated** and currently does not have a wired manifest editor (commit `71f70e8` ships the admin endpoints; the dashboard React app does not yet call `/v0/admin/manifest` or `/v0/admin/reconcile`). The CLI is the source of truth today.

### Via CLI `[PROTOCOL]`

1. **Edit the manifest on disk.** Append the node and any new `relationships` edges. Preserve existing edges — the manifest is the full declared state, not a diff.

2. **Add a runner script (only if the node is a long-lived process).** `[OPINIONATED]`
   - `scripts/run_<node_id>.sh` is the default lookup. Make it `chmod +x` and confirm it points at `$MESH_HOST:$MESH_PORT` and reads `$<NODE>_SECRET`.
   - Alternatively, declare the runner inline in the manifest:
     ```yaml
     metadata:
       runner:
         cmd: ["/usr/bin/env", "node", "dist/my-node.js"]
         env: { LOG_LEVEL: info }
         restart: permanent       # or transient | temporary
         max_restarts: 5
         restart_window_s: 60
     ```
     The script-resolver in `core/supervisor.py` reads `metadata.runner` first and falls back to `scripts/run_<id>.sh`.

3. **Set the new node's identity secret in the Core process environment.** Until Core is restarted, the new env var is not visible to the running supervisor. Two options:
   - (a) Restart Core. Cleanest, but drops every SSE stream — see §3 if you cannot afford that.
   - (b) Have the manifest declare a `identity_secret:` literal value (not `env:`) for the new node only. Acceptable for non-secret rotation events; treat the literal as a deployment artifact, not a long-lived choice.

4. **Push the manifest to Core.** `[PROTOCOL]`
   ```sh
   curl -fsS -X POST \
     -H "X-Admin-Token: $ADMIN_TOKEN" \
     --data-binary @manifests/<deploy>.yaml \
     http://$MESH_HOST:$MESH_PORT/v0/admin/manifest
   ```
   Core writes a `.bak` of the previous manifest, validates, and rolls back automatically on parse/load failure. A 200 means the new declared state is live.

5. **Reconcile to actually spawn the new process.** `[PROTOCOL]`
   ```sh
   curl -fsS -X POST -H "X-Admin-Token: $ADMIN_TOKEN" \
     http://$MESH_HOST:$MESH_PORT/v0/admin/reconcile
   ```
   Response shows `actions.spawned: [<new_id>]`. Anything in `actions.errors` or `actions.skipped` is the failure to investigate. `skipped: no_runner` means the resolver couldn't find `scripts/run_<id>.sh` and no inline runner was declared.

6. **Verify the new node registered.** Re-run `/v0/admin/processes` and `/v0/healthz`. The supervisor reports the pid; only registration confirms the protocol-level handshake completed.

### Via dashboard `[OPINIONATED]`

Currently aspirational. The endpoints exist; the React app does not yet wire a "Save manifest & reconcile" button. When that ships, the flow will be:
1. Edit manifest in the dashboard's manifest editor.
2. Click `Save & Reload` → POST `/v0/admin/manifest` then POST `/v0/admin/reconcile`.
3. Watch the `Processes` panel update from the SSE tap (`/v0/admin/stream` already emits `supervisor` events for spawn/stop/restart/exit/give_up).

Until then, do not hand-roll a UI flow that bypasses the audit-friendly two-step (write manifest, then reconcile). The atomicity story depends on it.

## 3. Rolling restart of one node without dropping envelopes

The supervisor's `restart` is a graceful stop followed by a spawn. It does **not** drain pending invocations — any in-flight `request_response` invocation routed to the node will see the SSE stream close and return a `denied_node_unreachable` or, if mid-call, a 504 timeout. To roll restart without dropping envelopes you must coordinate at the protocol layer, not just the OS layer.

### Procedure `[PROTOCOL]`

1. **Quiesce upstream callers.** `[OPINIONATED]`
   The protocol does not yet have a "drain" primitive. Use whatever the opinionated layer offers — for nodes that voluntarily report `node_status` (see `/v0/admin/node_status`), set `visible: false` and add a `details: {drain: true}` flag. Upstream nodes that respect the convention will stop sending new invocations.
   For nodes without a drain convention, gate upstream traffic by removing the relationship: hot-edit the manifest to drop the `from -> to` edge, POST it to `/v0/admin/manifest`, then `/v0/admin/reload` (which does not touch processes). New invocations will now hit `denied_no_relationship` at Core.

2. **Drain in-flight invocations.** `[PROTOCOL]`
   Watch `/v0/healthz` — `pending` is the count of `request_response` invocations Core is awaiting a response for. Wait for it to reach the steady-state (typically 0). If `pending` includes invocations *not* targeting this node, inspect `/v0/admin/state.envelope_tail` to identify them.

3. **Stop the node gracefully.** `[PROTOCOL]`
   ```sh
   curl -fsS -X POST -H "X-Admin-Token: $ADMIN_TOKEN" \
     -H "Content-Type: application/json" \
     -d '{"node_id":"<id>","graceful":true}' \
     http://$MESH_HOST:$MESH_PORT/v0/admin/stop
   ```
   The supervisor sends SIGTERM to the process group, waits 5 s, escalates to SIGKILL if needed.

4. **Spawn the new instance.** `[PROTOCOL]`
   ```sh
   curl -fsS -X POST -H "X-Admin-Token: $ADMIN_TOKEN" \
     -H "Content-Type: application/json" \
     -d '{"node_id":"<id>"}' \
     http://$MESH_HOST:$MESH_PORT/v0/admin/spawn
   ```
   Or, if you only need a fresh process with the same spec, use `/v0/admin/restart`, which does steps 3+4 atomically under the supervisor lock.

5. **Confirm re-registration.** `[PROTOCOL]`
   `/v0/healthz` `nodes_connected` returns to its prior value, and `/v0/admin/processes` shows the new pid `running`.

6. **Re-enable upstream traffic.** Reverse step 1 (set `visible: true` or restore the edge in the manifest and reload).

For zero-drop on `fire_and_forget` surfaces, the protocol provides no buffer — a deliver attempt to a disconnected node returns `503 denied_node_unreachable` to the sender. The opinionated layer must implement retry/queueing on the calling side. Don't ask the protocol to absorb that; it's deliberately out of scope.

## 4. Debugging: "a node stopped responding"

Symptoms typically arrive as `denied_node_unreachable` (503), invocation timeouts (504), or the dashboard showing the node as disconnected.

### Triage runbook

1. **Confirm Core itself is alive.** `[PROTOCOL]`
   `curl /v0/healthz`. If this fails, the problem is Core, not the node — jump to disaster procedures.

2. **Distinguish "process dead" from "process alive, registration lost".** `[PROTOCOL]`
   ```sh
   curl -H "X-Admin-Token: $ADMIN_TOKEN" /v0/admin/processes \
     | jq '.processes[] | select(.node_id=="<id>")'
   ```
   - `status: running` + `nodes_connected` view says the node *is* connected → it's responding to Core; the bug is application-level. Inspect the node's log under the supervisor `--supervisor-log-dir`.
   - `status: running` + the node is **not** in `nodes_connected` (from `/v0/admin/state`) → process is alive but its `register` call never succeeded. Causes: bad signature (mismatched `identity_secret`), wrong `MESH_HOST`/`MESH_PORT`, network partition. The audit log shows registration attempts; tail it.
   - `status: crashed` → supervisor saw an unexpected exit and is in backoff. `restart_count` and `last_exit_code` are on the response. The node will respawn on its own unless `restart_count > max_restarts`.
   - `status: failed` → supervisor gave up. Investigate logs before retrying. Then `POST /v0/admin/spawn` (which resets the restart-window counter on the next successful long-lived run).

3. **Read the node's log.** `[OPINIONATED]`
   `tail -200 /var/log/raven-mesh/<node_id>.log`. The supervisor brackets each lifecycle with `=== supervisor start at … ===` and `=== child exited rc=N at … ===` markers, so you can isolate the most recent run quickly.

4. **Replay the failing invocation.** `[PROTOCOL]`
   `/v0/admin/invoke` lets you synthesize a signed envelope from any registered node and replay the exact call. Pair it with `/v0/admin/stream` (SSE tap of every envelope through Core) in another terminal to see schema validation, edge ACL evaluation, and route status in real time.

5. **Check the audit log.** `[PROTOCOL]`
   `audit.log` is JSON-per-line. The `decision` field on `type: "invocation"` records (`routed`, `denied_unknown_node`, `denied_signature_invalid`, `denied_no_relationship`, `denied_unknown_surface`, `denied_schema_invalid`, `timeout`, `denied_node_unreachable`) is the canonical trace of why an invocation went where it did. Filter by `correlation_id` to see the request and response together.

6. **If the node is registration-flapping**, the loop is usually: process boots → registers → its identity secret rotates / its env breaks → next startup fails registration → supervisor restarts → repeat. The `restart_count` in `/v0/admin/processes` ticking up while `nodes_connected` stays low is the signature.

7. **Last resort — restart only the affected node.** `[PROTOCOL]` Use `/v0/admin/restart`. Do not restart Core; it would drop every connected node's SSE session and force the entire mesh through registration storm.

## 5. Backup + restore of the audit log

The audit log is **protocol-layer state**: it is the authoritative trace of every routing decision, signature verification, and admin action. Treat it like a write-ahead log.

### Backup `[PROTOCOL]`

The audit log is append-only JSON-per-line, written under an asyncio lock by `CoreState.audit()`. It never rotates itself.

- **Continuous shipping.** Tail-and-ship to your log aggregator. Each line is self-contained JSON with an `id` (UUIDv4) and `timestamp`, so dedup at the receiver is trivial. Do not ship via a wrapper that re-orders lines.
- **Periodic snapshot.** A point-in-time backup is a plain copy of the file. Because writes are line-oriented and the lock is per-write, a snapshot taken with `cp` (or `dd` to a different filesystem) may end on a partial line — the restore step trims it.
- **Cadence.** At minimum, snapshot before any rotation, before disaster recovery, and on the cadence required by your retention policy. There is no incremental/full distinction; the file is always the full history since last truncation.

### Rotation `[PROTOCOL]`

Out-of-process rotation is safe via `copytruncate`-style rotation:
```
/var/lib/raven-mesh/audit.log {
    daily
    rotate 30
    copytruncate
    compress
    missingok
    notifempty
}
```
Do **not** use rename-based rotation. `CoreState.audit_path` is resolved once at startup; renaming the file leaves Core writing to the renamed inode while the new path is empty. `copytruncate` keeps the inode stable.

### Restore `[PROTOCOL]`

Restore is rarely about replaying the log into anything — Core does not consume its own audit log on startup. Restore is for forensics and compliance. The procedure:

1. Stop Core (so nothing is appending). `[PROTOCOL]`
2. Place the desired prior log at `$AUDIT_LOG`.
3. Trim a partial trailing line if needed: `python3 -c 'import json,sys; [json.loads(l) for l in open(sys.argv[1])]' $AUDIT_LOG` and lop the final line if it raises.
4. Start Core. New entries append after the restored history.

If you need to *merge* a hot-shipped tail with a snapshot, sort by `timestamp` and dedup by `id` — both fields are populated on every entry by design.

## 6. Disaster: "the manifest got corrupted"

A corrupted manifest means Core cannot complete `load_manifest()`. Symptoms: `/v0/admin/manifest` returns `400 load_failed` (and Core auto-rolled back to `.bak`), or — worse — Core failed to start entirely.

### If Core is still running `[PROTOCOL]`

Core auto-rolls back on a failed `/v0/admin/manifest` POST. The `.bak` next to the manifest holds the last-known-good content, restored automatically. No operator action required *unless* the rollback itself failed (see logs for `error: load_failed` after the restore attempt).

### If Core failed to start `[PROTOCOL]`

The on-disk manifest is the only source of truth. Recovery options, in order of preference:

1. **Restore from the `.bak` sidecar.**
   ```sh
   ls -la manifests/<deploy>.yaml manifests/<deploy>.yaml.bak
   cp -p manifests/<deploy>.yaml.bak manifests/<deploy>.yaml
   python3 -c 'import yaml; yaml.safe_load(open("manifests/<deploy>.yaml"))'
   ```
   The `.bak` is the most recent successful `/v0/admin/manifest` write. It will not exist if the manifest has only ever been edited by hand (the auto-backup is created only by the admin endpoint). Treat this as the strongest argument for *only* changing manifests via `/v0/admin/manifest` in production.

2. **Restore from version control.**
   ```sh
   git log --oneline -- manifests/<deploy>.yaml | head
   git show <sha>:manifests/<deploy>.yaml > manifests/<deploy>.yaml
   ```

3. **Reconstruct from the audit log.** `[PROTOCOL]`
   `audit.log` records every `decision: routed` invocation with `from_node` and `to_surface`. From that you can recover the set of node IDs and the (from, to) edges that were definitely active. Schema files referenced by surfaces are *not* in the audit log — recover those from version control or backups.

4. **Reconstruct from supervisor state.** `[PROTOCOL]`
   If Core is up but the manifest on disk is wrong, `GET /v0/admin/state` returns the in-memory `nodes` and `relationships`. This is the manifest as Core last loaded it, including secrets resolved from env. You can dump it and re-emit a manifest YAML by hand, then write it back via `/v0/admin/manifest`.

### After restoration `[PROTOCOL]`

1. Start Core (or re-POST the manifest).
2. Run `/v0/admin/reconcile` — the running supervisor children will be diffed against the (now correct) manifest. Expect `kept` for every node that was already running with the right id; `spawned` for any whose id was missing during the corruption window; `stopped` for any orphaned children whose declarations are no longer present.
3. Smoke a known-safe invocation per §1 step 8.

### Prevention `[PROTOCOL]`

- Treat the manifest as deploy-controlled config. Every change goes through version control and through `/v0/admin/manifest` — do not hand-edit production manifest files in place. The endpoint's atomic write+validate+rollback is the safety net you want.
- Keep at least one off-host copy of the manifest *and* every JSON schema it references. Schemas are loaded relative to the manifest at registration time; losing a schema file is functionally equivalent to losing the manifest.
- Audit-log backup (§5) and manifest-source-control are independent disasters; treat them as such.

## Layer summary

| Action | Layer |
|---|---|
| `--supervisor`, `--auto-reconcile`, `/v0/admin/*` endpoints | Protocol |
| Manifest file content (which nodes, which edges) | Protocol declaration; specific values are deployment-specific |
| `scripts/run_<node_id>.sh` | Opinionated (one specific way to start one specific node) |
| `metadata.runner.cmd` in manifest | Protocol-level escape hatch; the value is opinionated |
| Audit log shape and write contract | Protocol |
| Audit log retention/rotation policy | Operational/opinionated |
| Dashboard wiring of admin endpoints | Opinionated |
| Drain conventions (`node_status.visible`, etc.) | Opinionated — protocol provides reporting, not policy |

When in doubt, prefer the admin API over hand-touching files. The endpoints exist precisely so the protocol layer can supervise its own state transitions; bypassing them shifts the contract back onto the operator.
